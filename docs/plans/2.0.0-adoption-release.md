# Nexus-MCP Adoption Fixes + 2.0.0 Release ‚Äî Plan

## Context

The repo is at version **2.0.0** in [pyproject.toml](../../pyproject.toml) with an accurate
[CHANGELOG.md](../../CHANGELOG.md) 2.0.0 entry, but **PyPI only has 0.1.0/0.1.1** ‚Äî v2.0.0 was
committed (`674a348`) yet never tagged or released. Every new adopter therefore `pip install`s
the old 15-tool server while all docs, the CLAUDE.md guidance, and the plugin skill describe the
consolidated 10-tool API (`graph`/`map`/`memory`). On top of that stale-vs-reality mismatch there
are three concrete adoption hurdles: a Python-version declaration that overstates support, a
`trust_remote_code=True` default the shipped model never needs, and the README's own
copy-paste boilerplate telling agents to call removed tools.

Goal: make a fresh `pip install nexus-mcp-ci` land on a coherent 2.0.0 where the docs, tool
names, server name, and config defaults all match the running server ‚Äî then publish it.

### Decisions locked with the user
1. **Python 3.13** ‚Üí restore the `<3.13` cap (match reality; tree-sitter-languages 1.10.2 has no 3.13 wheels).
2. **Canonical server name** ‚Üí **`nexus-mcp-ci`** (registration key + all `mcp__‚Ä¶__` tool prefixes). PyPI package/binary already `nexus-mcp-ci`; this makes the README top-line correct and moves everything else to match.
3. **Release** ‚Üí cut end-to-end: tag + create the GitHub release (triggers `publish.yml` ‚Üí PyPI via OIDC trusted publishing).
4. **trust_remote_code** ‚Üí flip default to `False`.

## Phase 4 ‚Äî Glama refresh

`glama.json` has **no version field** and a stale `configSchema`.
- Fix `NEXUS_EMBEDDING_MODEL` default `"jina-code"` ‚Üí `"bge-small-en"` (matches real default).
- Add missing env vars documented in README's config table (auto-watch, staleness interval, permission level, rate-limit, audit, trust_remote_code, log level/format).
- Add a `version: "2.0.0"` field **only if** the Glama `server.json` schema supports it (WebFetch the `$schema` URL to confirm). Glama otherwise re-derives version from PyPI after publish ‚Äî so the PyPI 2.0.0 release is the primary version signal.

## Phase 5 ‚Äî Verify, then release end-to-end

1. **Verify locally** (from a feature branch ‚Äî default branch requires branching first):
   - `pytest -v` (or `pytest -m "not slow"` for speed) ‚Äî all green.
   - `ruff check .` ‚Äî clean.
   - `python self_test/demo_mcp.py` ‚Äî all 10 tools pass end-to-end.
   - `python -m build` ‚Äî sdist+wheel build cleanly (mirrors the CI publish step).
2. **Land the changes**: commit on a feature branch, push, open PR, merge to `main`.
3. **Release**: tag `v2.0.0` on merged `main`, `gh release create v2.0.0` with CHANGELOG 2.0.0 notes ‚Üí `publish.yml` fires on `release: published` and publishes to PyPI via OIDC trusted publishing (`environment: pypi`, no local token).
4. **Confirm**: PyPI shows 2.0.0 (`pip index versions nexus-mcp-ci`); fresh `pip install nexus-mcp-ci` in a clean venv exposes the 10 consolidated tools; Glama re-scans to 2.0.0.

## Notes / risks
- `publish.yml` builds on Python 3.12 ‚Äî unaffected by the `<3.13` cap.
- Real Python 3.13 support (swapping tree-sitter-languages for tree-sitter-language-pack) is deliberately **out of scope** ‚Äî separate work item.
- Flipping `trust_remote_code` is behavior-changing only for jina-code users, who must now set `NEXUS_TRUST_REMOTE_CODE=true`; README/CLAUDE.md/config-table updated to say so.
